## 这是一款什么插件
简单来说，这是一款直接嵌入进Godot编辑器的全功能AI Agent插件，能对项目上下文自动进行感知，不需要第三方Agent，也不需要MCP。

> 本插件在早期完成了基础的功能开发后，就一直使用插件自身进行迭代，这意味本插件的工程实现来自真实经验，而不是单纯的套壳模仿。

## 安装
1. 下载本插件
2. 下载依赖插件： [Context Toolkit](https://github.com/snougo/Context-Toolkit) / [Godot Dom Parser](https://github.com/codeWonderland/godot-dom-parser) / [Godot Shader Library](https://github.com/Kelpekk/Godot-Shader-Library).
3. 将所有插件放入addons文件夹中
4. 启用本插件

> 注意：本插件虽然自身运行不依赖任何外部工具，但是少数供AI调用的工具依赖上述三个Godot插件暴露的API工作，你只需要下载这三个插件并放入addons插件文件夹中即可，无需启用。

## 插件设置
初次启用时，你需要找到插件的设置页面，并对 `Base Url`, `API Key` 以及 `System Prompt` 这三个设置选项进行填写。如果你不知道这三个设置选项是什么意思，可以直接发送截图给第三方AI询问，我就不在这里解释了。

### 可选设置
- **Tavily API Key**：配置后模型才可以正常使用 `search_web` 联网搜索工具。
- **Max Chat Turns**：默认 `20` 。控制发送给模型的对话轮数，过高可能消耗大量Token，过低则可能导致模型丢失对话早期的内容，推荐在对话过程中按需调整。
- **Temperature**：默认 `1.0` 。数值越低回答越严谨（适合写代码），数值越高回答越发散（适合创意）。
- **Log Level**：默认勾选了 `Info`, `Warn` 和 `Error`，一般不需要改动，除非你想要在编辑器的Ouput中实时查看插件的Debug输出。

> 如果你需要联网搜索功能，可以去Tavily官网申请一个免费的API Key。

> 插件仓库中有一份经过多轮迭代的默认系统提示词文档 `system_prompt.md`，开箱即用，推荐初次上手本插件的用户使用该系统提示词，然后根据实际需求再进行个性化调整。

> **注意**：模型厂商提供的 `Coding Plan` 服务通常只能用于它们指定的客户端，无法在本插件中使用，因此推荐直接使用Deepseek官方提供的API服务，目前该服务价格在命中缓存的情况下，已经非常便宜。或者你也可以使用本地部署的模型服务，比如LM Studio和ollama，

## 如何使用

<img width="349" height="100" alt="截屏2026-06-08 17 59 36" src="https://github.com/user-attachments/assets/14215aa4-cfff-4b41-8d81-d02b3aebf5a5" />

假设你成功进行了设置，那么你应该可以看到插件状态栏显示为蓝底白字的`Ready`状态，并且在模型切换框中看到多个模型可供选择。此时可以发送一个`你好` 作为测试，如果你已经充值了API，那么模型应该会马上回复你。

至于之后，你可以直接向模型询问本插件如何使用的问题，因为插件的默认工作区就位于本插件自身。

不过有一些使用技巧还是直接告诉你比较好：
- 每个会话尽量专注于同一个话题或者同一个问题，新问题或者新话题应该尽量新开会话，并设置正确的工作区。
- Godot游戏项目通常文件众多，因此良好的项目文件管理和正确的工作区设置可以让模型输出更准确。
- 本插件并未启用所有的AI工具，因此你可以根据自己需要将其添加进 `tool_registry.gd` 脚本中的 `CORE_TOOLS_PATHS` 常量中，不过更推荐的做法是将工作流复用为Skill，详情可以直接通过本插件向AI询问如何制作自定义Skill。
- 本插件支持向多模态模型发送图片内容，你可以通过部署一个本地的小参数多模态模型来体验这个特性。
- 本插件的用户输入框支持向里面拖拽文件，当拖拽完毕后，会自动将其处理成文件路径，当你点击发送按钮后，插件会根据文件类型自动将文件路径转换成对应的内容。

总之，如果你对本插件有什么疑问，可以直接通过本插件向AI进行询问，我不再作过多的解释了。

## 新手上手练习
以下练习全部基于插件的默认工作区（即插件自身）。因为插件目录被限制为只读，你可以放心尝试任何操作——它只能阅读和解释，不会改坏任何文件。

第一步：确认连接
发送 `你好`。 如果看到模型回复，说明 API 配置正确。

第二步：让它“看懂”这个插件
发送 `插件有哪些模块？` ，然后观察模型是如何回应的。

第三步：让它查到你不熟悉的API
发送 `这个插件的流式请求是怎么处理的？`，然后观察模型是如何回应的。

第四步：体验"全局追踪"
发送 
```
帮我在整个插件中搜索对 ChatMessage.ROLE_TOOL 的引用，
然后告诉我这条消息的完整生命周期 —— 从创建、渲染到发送给 API
```
然后观察模型是如何回应的。

第五步：体验多模态
随便找张图片拖拽进用户输入框中，然后添一句话 `描述一下图片内容`，点击发送，观察模型的回应（模型需要支持多模态，否则会报错）。
