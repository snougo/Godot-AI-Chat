## Runtime Debug SOP

### 概览
本技能指导如何正确使用 **运行时错误捕获工具链** 来调试游戏中的 GDScript 运行时错误。
核心工作流为：**启动捕获 → 运行场景 → 获取错误 → 分析修复 → 停止捕获**。

### 触发条件
当用户报告游戏出现运行时错误（如 null 访问、数组越界、类型错误等），或用户要求调试/排查游戏中的 bug 时。

### 指令

#### 调试流程（必须按顺序执行）

**第一步：启动错误捕获**
- 调用 `capture_runtime_errors("start")`
- 目的：启用文件日志系统，准备记录运行时错误
- 如果返回 "Already capturing"，先调用 `clear` 再重新 `start`

**第二步：运行测试场景**
- 调用 `run_scene(scene_path="res://path/to/scene.tscn")`
- 目的：运行目标场景，触发可能的运行时错误
- 场景路径由用户指定或根据上下文推断

**第三步：等待错误出现并停止**
- 让场景运行片刻，等待错误触发
- 调用 `stop_game()` 停止游戏运行

**第四步：获取捕获到的错误**
- 调用 `capture_runtime_errors("get_errors")`
- 目的：读取日志文件中新增的错误信息
- 如果返回 "No errors captured"，可能原因：
  - 场景未触发运行时错误
  - 文件日志尚未写入（可稍后重试）

**第五步：分析错误并修复**
- 根据返回的错误信息（错误类型、位置、堆栈），分析根因
- 使用 Script Tool SOP 技能修复代码

**第六步：清除错误记录（可选）**
- 调用 `capture_runtime_errors("clear")`
- 目的：清除已捕获的错误，为下一轮调试做准备

#### 参数详解

| 工具名称 | 参数 | 说明 |
|---------|------|------|
| `capture_runtime_errors` | action: "start" / "stop" / "get_errors" / "clear" | 错误捕获生命周期管理 |
| `run_scene` | scene_path (String) | 运行指定场景文件（.tscn / .scn） |
| `stop_game` | - | 停止当前正在运行的游戏 |

#### 最佳实践
1. **完整闭环**：每次调试必须走完 `start → run → stop → get_errors` 完整流程
2. **先清后启**：如果上次捕获未清理，先 `clear` 再 `start`，避免新旧数据混淆
3. **增量读取**：`get_errors` 只返回自上次调用以来的新增错误，可多次调用
4. **错误定位**：返回的错误包含文件名和行号，可直接用于定位修复
5. **一次一错**：GDScript 遇到第一个错误就会停止执行，因此一次调试可能只捕获一个错误，修复后需重新运行测试
